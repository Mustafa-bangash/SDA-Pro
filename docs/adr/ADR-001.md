# ADR-001: SOA vs Microservices vs Modular Monolith

## Status
Accepted

## Context
SDA-Pro needs a distributed architecture to handle alert ingestion,
enrichment, incident management, and response orchestration.
We evaluated three options:
- Microservices
- Modular Monolith
- Service-Oriented Architecture (SOA)

## Decision
We chose SOA because:
- Each core capability (Ingestion, Enrichment, Incident Management,
  Response, Threat Intel, Notification, Audit) maps naturally to
  an autonomous service with a well-defined contract.
- Unlike pure Microservices, SOA allows shared canonical data models
  (Alert, Incident) across services via the shared/ library,
  reducing duplication without tight coupling.
- Unlike Modular Monolith, SOA supports independent deployment
  and scaling of high-load services like Alert Ingestion.

## Consequences
- Services communicate via REST/gRPC (synchronous) and
  RabbitMQ (asynchronous/event-driven).
- Each service has its own database (PostgreSQL for incidents,
  Redis for caching).
- Integration complexity is managed via well-defined API contracts
  in docs/api/.
