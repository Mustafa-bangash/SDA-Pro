# ADR-001: SOA vs Microservices vs Modular Monolith

## Status
Accepted

## Context
SDA-Pro needs a distributed architecture to handle multiple responsibilities including
alert ingestion, enrichment, incident management, response orchestration, threat intel
integration, notifications, and audit logging. We evaluated three architectural approaches:

**Option 1: Microservices**
- Each tiny function gets its own service
- Maximum flexibility but very high operational overhead
- Difficult to manage for a 3-person team in 2 weeks
- Requires service mesh, distributed tracing, complex networking

**Option 2: Modular Monolith**
- All modules in one deployable unit
- Simple to develop and deploy initially
- Cannot scale individual components independently
- Alert ingestion load would affect dashboard performance
- Violates the requirement for autonomous service boundaries

**Option 3: Service-Oriented Architecture (SOA)**
- Coarse-grained autonomous services grouped by business capability
- Shared canonical data models via shared/ library
- Each service independently deployable and scalable
- Well-defined contracts between services
- Manageable complexity for a 3-person team

## Decision
We chose SOA because:

1. Each core capability maps naturally to an autonomous service with a clear boundary.
   Alert ingestion, enrichment, incident management, and response orchestration are
   distinct business domains with different scaling needs.

2. Unlike pure Microservices, SOA allows shared canonical models (CanonicalAlert,
   Incident) across services via the shared/ library, reducing duplication without
   creating tight coupling between services.

3. Unlike Modular Monolith, SOA supports independent deployment and scaling.
   The alert-ingestion-service can be scaled horizontally during high-volume attack
   scenarios without affecting the soc-dashboard or incident-management-service.

4. SOA fits the team structure naturally. Each student owns 2-3 services aligned
   to their pattern responsibilities, making parallel development possible.

## Consequences

**Positive:**
- Services communicate via REST/gRPC for synchronous calls and RabbitMQ for
  asynchronous event-driven communication.
- Each service has its own database (PostgreSQL for incidents, Redis for caching).
- Integration complexity is managed via well-defined API contracts in docs/api/.
- New alert sources can be added to alert-ingestion-service without touching
  enrichment or incident management services.

**Negative:**
- Distributed system adds network latency between services.
- Requires Docker Compose to run the full stack locally.
- Integration testing is more complex than a monolith.
- Team must agree on canonical event schemas in shared/events/.
