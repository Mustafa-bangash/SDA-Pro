# ADR-004: Threat Intel Caching Strategy

## Status
Accepted

## Context
SDA-Pro integrates with external threat intelligence providers like VirusTotal and MISP
to check reputation of IP addresses, domain names, and file hashes found in alerts.

The problem is:
- VirusTotal free tier allows only 4 requests per minute and 500 per day
- The same IP address may appear in hundreds of alerts during a single attack campaign
- Without caching, a single attack with 500 alerts hitting the same malicious IP
  would exhaust the daily API quota in minutes
- External API calls add 200-800ms latency to the enrichment pipeline

We evaluated four caching options:

**Option 1: No Caching**
- Simple but completely impractical
- API quota exhausted immediately during real attacks
- High latency on every enrichment request

**Option 2: In-Memory Cache (Java HashMap)**
- Fast and simple to implement
- Cache is lost on service restart
- Cache is not shared between multiple instances of enrichment service
- No TTL management — stale threat intel stays forever

**Option 3: Redis Cache**
- Shared across all service instances
- Survives service restarts
- Built-in TTL expiry for automatic cache invalidation
- Supports atomic operations for cache stampede prevention
- Slightly more complex to set up than in-memory

**Option 4: External Cache Service (Memcached)**
- Similar to Redis but fewer features
- No persistence, no pub/sub, no data structures
- Less useful for other purposes in the project

## Decision
We chose Redis as the caching layer, implemented via the Proxy pattern
(ThreatIntelProxy) because:

1. Redis TTL perfectly models threat intel validity windows. A reputation score
   for an IP address cached for 1 hour balances freshness vs API cost.

2. The Proxy pattern wraps the real ThreatIntelProvider and transparently serves
   cached results. Client code does not know whether a result came from cache or
   the real API. This is exactly the Proxy pattern justification.

3. Redis is already in the stack for session management (ADR-003), so no new
   infrastructure is added.

4. Cache invalidation is handled via the ThreatIntelUpdated event. When MISP
   pushes new threat intel, the event triggers cache invalidation for affected
   indicators.

## Cache TTL Strategy
| Indicator Type | TTL | Reason |
|----------------|-----|--------|
| IP Address reputation | 1 hour | IPs can be reassigned quickly |
| Domain reputation | 6 hours | Domains change ownership slowly |
| File hash verdict | 24 hours | Hash verdicts rarely change |
| Known clean indicators | 12 hours | Avoid re-checking known safe IPs |

## Consequences

**Positive:**
- API quota is preserved. Repeated checks for same indicator hit Redis not VirusTotal.
- Enrichment pipeline latency drops from 500ms to under 5ms on cache hits.
- Cache is shared across all enrichment service instances automatically.
- TTL ensures threat intel stays reasonably fresh without manual invalidation.

**Negative:**
- A newly malicious IP that was cached as clean will be missed until TTL expires.
- Redis must be running for enrichment to work at full speed.
- Cache key design must handle indicator normalization (lowercase domains, etc).
