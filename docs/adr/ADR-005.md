# ADR-005: Real-Time Push Strategy — WebSockets vs SSE vs Polling

## Status
Accepted

## Context
The SOC Analyst Dashboard needs to display real-time updates when:
- A new alert is ingested and enriched
- An incident changes state (New to UnderTriage, Containment, etc.)
- A response action is executed (IP blocked, endpoint isolated)
- New threat intel arrives for an active incident

The dashboard must update in under 2 seconds as per acceptance criteria.

We evaluated three approaches:

**Option 1: Client Polling**
- Dashboard sends GET request every N seconds to check for updates
- Simple to implement, works everywhere
- Wastes bandwidth — most polls return no new data
- Minimum latency equals the polling interval
- Does not scale well with many analysts logged in simultaneously
- Cannot meet the less than 2 second update requirement reliably

**Option 2: Server-Sent Events (SSE)**
- Server pushes updates to client over a single long-lived HTTP connection
- One-directional — server to client only
- Built into browsers natively, no extra libraries needed
- Automatically reconnects if connection drops
- Works well for alert streams and incident state change notifications
- Cannot receive messages from client over same connection

**Option 3: WebSockets**
- Full duplex — both server and client can send messages
- Lower overhead per message than HTTP
- Requires WebSocket server infrastructure
- Slightly more complex than SSE
- Best for interactive two-way communication like analyst annotations
- Supported by Spring Boot via STOMP over WebSocket

**Option 4: Hybrid SSE + REST**
- SSE for server-to-client push notifications
- Regular REST for client-to-server actions
- Simpler than full WebSockets for mostly read-heavy dashboard

## Decision
We chose WebSockets (STOMP over Spring WebSocket) as the primary real-time
mechanism because:

1. The SOC dashboard is not purely read-only. Analysts trigger response actions,
   annotate incidents, and override automated decisions — all requiring
   client-to-server messages. WebSockets handle this naturally.

2. Spring Boot has excellent WebSocket support via spring-websocket and STOMP,
   making implementation straightforward.

3. The Observer pattern maps directly onto WebSocket subscriptions. When
   EventBusPublisher notifies DashboardUpdater, DashboardUpdater pushes to all
   subscribed WebSocket sessions immediately.

4. SSE is kept as a fallback for environments where WebSockets are blocked by
   corporate firewalls (common in enterprise SOC environments).

## Implementation Mapping to Observer Pattern
- EventBusPublisher (Singleton) receives domain events from RabbitMQ
- DashboardUpdater (Observer) subscribes to relevant event types
- DashboardUpdater holds active WebSocket sessions map
- On event received, DashboardUpdater pushes to all connected analyst sessions
- Analysts subscribe to topics: /topic/alerts, /topic/incidents, /topic/actions

## Consequences

**Positive:**
- Dashboard updates in under 500ms after an incident state change.
- Meets the less than 2 second update acceptance criteria comfortably.
- Analysts see live alert stream without refreshing the page.
- Observer pattern decouples event publishing from dashboard delivery completely.

**Negative:**
- WebSocket connections are stateful and consume server memory per analyst session.
- Load balancer must support sticky sessions or shared session store for WebSockets.
- Corporate firewalls sometimes block WebSocket upgrades requiring SSE fallback.
