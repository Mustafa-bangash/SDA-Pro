# ADR-003: Database Strategy — PostgreSQL + Redis

## Status
Accepted

## Context
SDA-Pro needs to store several different types of data with different characteristics:

**Incident and Alert Data:**
- Incidents have complex relationships (alerts, states, actions, analysts)
- Need ACID transactions for state transitions (cannot have partial updates)
- Need complex queries (filter by severity, state, date range, analyst)
- Audit trail requires immutable ordered records

**Raw Alert Payloads:**
- Alerts come in different formats from different sources (Splunk JSON,
  CrowdStrike JSON, Firewall syslog, etc.)
- Schema varies by source type — no fixed columns
- High volume — thousands of alerts per hour during attack scenarios
- Mostly write-heavy with occasional reads for enrichment

**Threat Intel Cache:**
- Reputation checks for same IP/domain/hash repeated many times
- External API calls are expensive (rate limited, paid per query)
- Cache needs to expire automatically (threat intel changes over time)
- Must be shared across all enrichment service instances

**Session and Rate Limiting:**
- Analyst sessions need fast read/write with automatic expiry
- API rate limit counters need atomic increment operations

We evaluated three options:
- PostgreSQL only
- MongoDB only
- PostgreSQL + Redis hybrid

## Decision
We chose PostgreSQL for relational data and Redis for caching because:

1. **PostgreSQL** handles incidents, alerts, audit logs, and analyst data perfectly.
   ACID compliance ensures incident state transitions are atomic. Complex JOIN
   queries for incident reporting and compliance evidence are well supported.
   Row-level security can enforce analyst access controls.

2. **Redis** handles threat intel caching with automatic TTL expiry. A cache hit
   avoids an expensive VirusTotal API call. Redis atomic operations handle rate
   limit counters safely across multiple service instances. Session tokens with
   auto-expiry are a natural fit for Redis.

3. We considered MongoDB for raw alert payloads due to schema flexibility, but
   keeping everything in PostgreSQL with a JSONB column for raw_payload gives us
   flexibility without adding a third database technology to manage.

## Database Assignment Per Service
| Service | Database | Reason |
|---------|----------|--------|
| alert-ingestion-service | PostgreSQL | Store normalized alerts with JSONB raw payload |
| enrichment-correlation-service | PostgreSQL | Store enrichment results and incident clusters |
| incident-management-service | PostgreSQL | Store incidents with full state history |
| threat-intel-service | Redis | Cache reputation scores with TTL |
| identity-service | PostgreSQL + Redis | User data in PG, sessions in Redis |
| audit-service | PostgreSQL | Immutable append-only audit log |

## Consequences

**Positive:**
- Single relational database technology for most services reduces operational burden.
- JSONB in PostgreSQL handles variable alert formats without schema migration.
- Redis TTL handles threat intel cache expiry automatically.
- Redis atomic counters handle rate limiting correctly under concurrent load.

**Negative:**
- Raw alert JSONB columns in PostgreSQL are less efficient than a document store
  for very high volume scenarios above 100k alerts per hour.
- Redis data is lost on restart unless persistence is configured (AOF/RDB).
- Team must manage two database technologies in Docker Compose.
