# ADR-002: Synchronous vs Asynchronous Inter-Service Communication

## Status
Accepted

## Context
SDA-Pro services need to communicate with each other. We evaluated three approaches:

**Option 1: Fully Synchronous (REST only)**
- Simple to implement and debug
- Every service call waits for a response
- Alert ingestion would be blocked waiting for enrichment to finish
- Single failure cascades and brings down the whole pipeline
- Dashboard would need to poll constantly for updates

**Option 2: Fully Asynchronous (Message Bus only)**
- No service waits for another
- Very complex to implement for request-response patterns
- Difficult to get immediate feedback for analyst actions on dashboard
- Hard to handle errors and rollbacks

**Option 3: Hybrid Approach (REST + RabbitMQ)**
- Synchronous REST/gRPC for request-response patterns that need immediate results
- Asynchronous RabbitMQ events for fire-and-forget notifications
- Best of both worlds

## Decision
We chose the Hybrid approach because different communication needs exist in SDA-Pro:

**Synchronous (REST/gRPC) is used when:**
- Dashboard queries incident list from incident-management-service
- Response orchestration checks threat intel reputation scores
- Analyst triggers a manual response action and needs immediate confirmation
- Identity service validates analyst credentials

**Asynchronous (RabbitMQ Events) is used when:**
- Alert ingestion publishes AlertIngested event after normalizing
- Incident management publishes IncidentStateChanged after state transition
- Response orchestration publishes ResponseActionExecuted after completing
- Dashboard, Audit, Notification services all subscribe to relevant events
- No service waits for dashboard or audit to finish before continuing

## Event Schema (Observer Pattern Implementation)
| Event | Publisher | Subscribers |
|-------|-----------|-------------|
| AlertIngested | alert-ingestion-service | enrichment, dashboard, audit |
| AlertEnriched | enrichment-correlation-service | correlation, dashboard, audit |
| IncidentCreated | incident-management-service | dashboard, notification, response |
| IncidentStateChanged | incident-management-service | dashboard, audit, metrics |
| ResponseActionExecuted | response-orchestration-service | dashboard, audit, notification |

## Consequences

**Positive:**
- Alert ingestion pipeline is non-blocking. High alert volume does not slow
  down the dashboard or incident management.
- Dashboard gets real-time push updates via WebSocket without polling.
- Audit service receives all events independently without being in the critical path.
- Loose coupling between services via the Observer pattern on the event bus.

**Negative:**
- Developers must handle eventual consistency between services.
- Message broker (RabbitMQ) is an additional infrastructure component.
- Debugging async flows requires distributed tracing tooling.
